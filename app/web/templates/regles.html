{% endmacro %} {% call sec('s0', '0 · Séquence de clôture (6 étapes)', true) %} Ordre d'exécution, par établissement : 1. Synchronisation des clients → 2. Génération du CSV (caisse + factures, un seul fichier) → 3. Import du CSV dans Pennylane (manuel) → 4. Cadrage de l'import par Vaelan (vérification API) → 5. Attache des justificatifs (PDF factures) → 6. Lettrage. Le tableau de suivi (page d'accueil société) trace « jusqu'à quelle date » et « réalisé le… » par étape. Couleur : vert si couvert jusqu'au dernier mois TERMINÉ (fin du mois précédent), sinon orange « mois à clôturer ». {% endcall %} {% call sec('s1', '1 · Synchronisation des clients') %} Table de correspondance TopOrder ↔ Pennylane. Clé de matching = SIRET TopOrder ↔ « Identifiant client » Pennylane (champ external_reference, où l'on met le SIRET ; PAS le SIREN qui est partagé entre établissements). Cascade de statut stricte : pas de SIRET → SIRET manquant ; SIRET sans jumeau Pennylane → absent de Pennylane ; jumeau sans compte 411 → incohérent ; jumeau + 411 → ok. Une ligne du mapping introuvable dans TopOrder → absent TopOrder. Vaelan détecte ; la correction se fait dans TopOrder/Pennylane puis on resynchronise. Les identifiants assignés (asso « ASDR ») sont gérés. {% endcall %} {% call sec('s2', '2 · Génération du CSV (caisse + factures, fichier unique)') %} Un seul CSV contenant deux journaux (Pennylane les sépare par la colonne « Code journal ») : TOSLT (caisse) et TOSLF (reclassement). Le ticket est la vérité fiscale ; tout le CA va au 70101 (= le Z).
  • Tickets anonymes → 1 écriture agrégée par jour : crédit 70101/taux + TVA collectée ; débit encaissements (CB/Espèce/TR, rendu monnaie netté) ; écart de caisse résiduel.
  • Tickets facturés → 1 écriture par facture : crédit 70101/taux + TVA ; débit 411 client du brut TTC (lettrage F<n°> = la facture) ; par paiement : débit encaissement + crédit 411 (règlement, lettré). Solde 411 restant = impayé.
Format CSV validé import Pennylane : comptes par NUMÉRO, journal par CODE (TOSLT), taux en virgule (2,1%), catégorie analytique sur TOUTES les lignes (Etablissement / nom court). Libellé de pièce « SM280526 #SMT03 » (jour en tête, lot en queue). Identifiant de lot supprimable d'un bloc. Noms de fichiers préfixés AAAAMMJJ HHMM (heure Réunion). {% endcall %} {% call sec('s2b', '2b · Pré-vol & cadrage (verrou de génération)') %} Pré-vol client (souple) : la génération est refusée uniquement si une facture de la période ne se route vers aucun compte 411 ; on liste les clients à corriger.
Cadrage : on compare le CA recalculé sur les tickets à la synthèse PDF. Le CA Total (et la période) sont BLOQUANTS. Un écart par mode de paiement (ex. CB) vient des tickets facturés payés comptant que la synthèse TopOrder classe hors « caisse ». Vaelan les RÉCONCILIE : il vérifie que l'écart est entièrement couvert par les paiements de factures encaissés en caisse (bookés au crédit du 411, lettrés F<n°>). Si oui → contrôle « COUVERT » (l'écart ne vient pas de nulle part, il est traité dans le CSV). Si l'écart dépasse les paiements de factures → avertissement « non expliqué ». Tolérance ± 0,05 €. Compte rendu PDF : Synthèse | API tickets | CSV agrégé par poste + section « Réconciliation des écarts de paiement ». {% endcall %} {% call sec('s4', '4 · Cadrage de l\'import par Vaelan (vérification Pennylane)') %} Re-jouable à tout moment. Vaelan relit les écritures des deux journaux (TOSLT + TOSLF) dans Pennylane (via l'API) et compare l'Attendu (CSV générés et stockés) à Pennylane. Le compte rendu PDF détaille tous les contrôles : CA HT / TVA / CA TTC, comptes clients 411, paiements par mode, réconciliation des paiements de factures (les paiements de factures bookés au crédit du 411 — ceux qui expliquaient l'écart de mode de paiement — sont confirmés présents dans Pennylane), et le détail par compte. Détecte toute suppression / modification / ajout depuis l'import. Horodate la dernière vérification (« c'était bon à cette date »). {% endcall %} {% call sec('s2c', 'Factures (TOSLF) & règlements — dans le même CSV') %} Reclassement (journal TOSLF, par facture) : débit 70101 / crédit 7012 (B2B) ou 70102 (B2C), HT par taux (avoirs inversés) → le CA facturé sort du 70101, net nul sur le CA total, pas de double comptage.
Écriture facture (journal TOSLT) : CA 70101/TVA + débit 411 brut (la facture) + paiements comptant (encaissement + crédit 411), lettrés F<n°>. Solde 411 = impayé.
Paiements décalés : une facture peut être réglée en caisse plus tard. Deux flux par période — CA par date de ticket ; règlements par date de paiement (liaison facture↔paiement via orderPaymentId), débit encaissement / crédit 411 F<n°>, lettrés à travers les périodes. {% endcall %} {% call sec('s8', '5 · Justificatifs (PDF factures) — par Vaelan via l\'API') %} Les écritures sont créées par import CSV manuel (sans pièce jointe). À l'étape 5, Vaelan relit les écritures « facture » dans Pennylane (pièce contenant F<n°>, journaux TOSLT créance + TOSLF reclassement), télécharge le PDF officiel de chaque facture depuis TopOrder (pdf.toporder.fr/pdf?InvoiceId=…, le GUID fait jeton, sans auth) et le rattache à l'écriture existante via l'API (POST /file_attachments puis PUT /ledger_entries/{id} avec file_attachment_id). C'est idempotent : une écriture déjà pourvue d'une pièce est ignorée. L'étape finit par une vérification dans Pennylane (champ attachment par écriture) et un rapport de complétude : nombre d'écritures facture, déjà attachées, attachées ce passage, en échec (avec la raison), couvertes au final, et le détail par facture. Vert « complet » uniquement si 0 écriture facture sans PDF. {% endcall %} {% call sec('s9', '6 · Lettrage des comptes 411 — par Vaelan via l\'API') %} Étape au niveau société (les comptes clients 411 sont partagés par les 3 boulangeries), lançable à tout moment avec une date d'arrêté (« lettrer jusqu'au … »). Vaelan lit chaque compte client tous journaux confondus (filtre par ledger_account_id) — il capte donc aussi les virements du journal de banque — et ne garde que les lignes ouvertes (lettered_ledger_entry_lines.ids vide). Il regroupe par facture (réf. F<n°>) et lettre via POST /ledger_entry_lines/lettering : soldé → lettrage complet ; acompte → lettrage partiel ; virement (crédit sans réf.) rapproché seulement si certain (une seule créance ouverte du même montant) ; ambigu / impayé → laissé ouvert et listé. Idempotent : on relit tout l'ouvert à chaque passe (un virement arrivé après coup est capté à la passe suivante — aucune hypothèse sur l'absence passée). Compte rendu : factures soldées lettrées, lettrages partiels (reste dû), virements rapprochés, ambigus à traiter à la main, créances impayées avec ancienneté. {% endcall %} {% call sec('s10', '7 · Suivi des paiements (page continue)') %} Page autonome (pas une étape de clôture séquentielle) de rapprochement TopOrder ↔ Pennylane par client (clé = compte 411, company-wide). Un job de synchro calcule un instantané par client : encours facturé impayé (solde 411 ouvert côté Pennylane), en attente de facturation (commandes TopOrder différées non facturées — statuts WAITING_FOR_BILLING / PURCHASE_ORDER —, mois courant / antérieur), exposition totale (= facturé impayé + en attente, tri par défaut décroissant), et l'écart TopOrder − Pennylane. L'API TopOrder ne permettant pas de saisir les règlements reçus, le détail d'un client liste ses virements (crédits du journal de banque) à reporter manuellement dans TopOrder : une case à cocher trace qui l'a reporté et quand (clé = ligne Pennylane, idempotent). Recherche, filtres (écart / à reporter / encours), tri colonnes, pagination.
Détail client = grand-livre complet du compte 411 Pennylane : toutes les écritures (date, libellé, journal, débit, crédit, état de lettrage) avec solde courant ligne à ligne + totaux et solde final. Chaque écriture est tagable : Saisi (reportée / rapprochée dans TopOrder) ou Ignoré (non pertinente / déjà traitée), avec traçabilité qui & quand (table PaymentReport, clé = ligne Pennylane). À venir : le côté TopOrder (factures, paiements, avoirs) en regard pour rapprocher les deux comptes ligne à ligne.
Bascule Kimayo → TopOrder : avant TopOrder, la facturation passait par Kimayo. Pour chaque client, le cutover = date de sa 1re facture TopOrder (plus ancienne créance F<n°>). On ne compte que les créances et virements à partir de cette date : les créances Kimayo antérieures sont exclues, et un virement reçu avant la 1re facture TopOrder n'est jamais retenu (on ne peut pas régler une facture pas encore émise). Règle métier : après bascule, plus de Kimayo. {% endcall %} {% call sec('cfg', 'Configuration & traçabilité') %} Tous les comptes (70101/70102/7012, TVA par taux, encaissements par établissement), journaux (code + id) et catégories analytiques sont éditables (page Configuration) ; seules les valeurs modifiées sont stockées. Chaque tâche conserve : la synthèse fournie, le CSV généré, le compte rendu PDF, la version de Vaelan et l'utilisateur. Tâches relançables ; suivi réinitialisable par établissement. {% endcall %}
